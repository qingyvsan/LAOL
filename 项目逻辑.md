

### 模块一：数据模型定义（任务与锁的结构）



为了让逻辑清晰，定义两个核心数据结构：



**1. 任务定义 (**`tasks/task_{uuid}.json`**)**

`json`

{

  "id": "uuid",

  "status": "pending\|in*progress\|done\|failed",*

  "description": "修改 auth.ts 的登录逻辑",

  "target*files": ["src/auth.ts", "src/utils/validator.ts"],*

  "assigned*agent": null,*

  "created*at": 123456,*

  "updated*at": 123456,*

  "dependency": null  // 可选的串行依赖ID

}





**2. 文件租约定义 (**`locks/{file_path}.lock`**)**

`json`

{

  "file": "src/auth.ts",

  "holder": "agent-001",

  "task*id": "uuid",*

  "expires*at": 123456  // 关键：TTL 机制防死锁*

}

### 模块二：中央调度器逻辑（`scheduler.py` 或 `scheduler.js`）



这是系统的大脑，负责**发现任务**和**分配租约**。它可以作为一个常驻的轻量级 CLI 进程运行。



`python`

# 伪代码 - 调度器核心循环 (每 10 秒执行一次)

while True:

    # 1. 清理过期租约 (防死锁)

    for lock in list*locks():*

        if lock.expires*at \< now():*

            release*lock(lock.file)  # 强制释放*

            notify*agent(lock.holder, "你的租约已过期")*



    # 2. 扫描待处理任务

    pending*tasks = scan*tasks(status="pending")

    

    for task in pending*tasks:*

        # 3. 冲突预检：检查任务涉及的任意文件是否被锁定

        conflicts = [f for f in task.target*files if is*locked(f)]

        

        if conflicts:

            # 若有冲突，将任务状态回退为 pending 并标记等待

            log(f"任务 {task.id} 等待文件: {conflicts}")

            continue

        

        # 4. 分配任务：为所有目标文件签发租约

        agent*id = get*idle*agent()  # 简单轮询调度*

        for f in task.target*files:*

            create*lock(file=f, holder=agent*id, ttl=300)  # 5分钟租约

        

        # 5. 更新任务状态，分配给 Agent

        task.status = "in*progress"*

        task.assigned*agent = agent*id

        save*task(task)*

        

        # 6. 调用子进程：派发 Agent 执行

        subprocess.Popen(["claude", "--agent", agent*id, "--task", task.id])*

### 模块四：实时感知与防冲突的关键算法



你担心的“Agent不知道别人在改什么”，通过以下**合并感知算法**解决，这发生在调度器分配任务时（**主动防御**），而不是修改发生后才察觉。



`python`

# 冲突检测算法 (在调度器分配任务时执行)

def can*assign*task*safely(task):*

    # 1. 检查直接冲突 (同一文件)

    for file in task.target*files:*

        if is*locked(file):*

            return False, f"{file} 正被占用"

    

    # 2. 检查逻辑冲突 (同一模块下的不同文件)

    # 如果两个任务分别改 auth.ts 和 auth-utils.ts，逻辑上紧密耦合

    module*name = extract*module(task.target*files[0])*

    other*locks = get*locks*in*module(module*name)*

    if other*locks:*

        # 策略: 不直接拒绝，而是打上 "高风险" 标签

        task.risk*level = "high"*

        notify*scheduler*ui(f"注意：{module*name} 模块正被 {other*locks.holder} 修改")

    

    return True, "分配成功"

### 模块五：智能合并策略（最终防线）



当两个 Agent 各自完成工作并提交 PR 时，采用 **Git 自定义合并驱动** 解决最终冲突。



**1. 注册 **`.gitattributes`：



src/**/***.ts merge=ai-merge*





**2. 合并驱动脚本 (**`git-merge-llm`**) 逻辑**：

`python`

def ai*merge(base, ours, theirs):*

    # 1. 首先尝试 Git 常规逐行合并

    result = subprocess.run(["git", "merge-file", ours, base, theirs])

    

    if result.returncode == 0:

        # 无冲突，直接通过

        return 0

    

    # 2. 如果有冲突，调用 LLM 理解意图

    prompt = f"""

    这是冲突文件。Base版本是...，Agent A 修改了...，Agent B 修改了...

    如果两者功能独立，请合并保留两者；如果逻辑矛盾，请只保留更合理的一方。

    输出最终合并代码：

    """

    resolved*code = call*llm(prompt)

    write*file(ours, resolved*code)

    

    # 3. 如果 LLM 也无法解决，返回非 0 退出码，暂停等待人工介入
