## 优化一：完全事件驱动替代轮询（P0，延迟 10s → <100ms）

**问题**：调度器 10 秒轮询一次 `tasks/`，新任务到达平均等 5 秒才被领取。Socket 只覆盖了"锁释放"，未覆盖"新任务到达""任务状态变更""heartbeat 超时"等事件。

**方案**：调度器不再定时循环，改为纯事件驱动。

```
# 调度器事件循环（Python 伪代码）

watcher = Watchdog([
    (".multiagent/tasks/",   on_task_created),     # 新任务即刻分配
    (".multiagent/locks/",   on_lock_changed),     # 锁变化即刻推送
    (".multiagent/staging/", on_intent_created),    # 意图文件出现即刻处理
])

scheduler = EventScheduler(
    socket_path=".multiagent/event.sock",
    watchers=watcher
)

# 事件类型映射
EVENT_HANDLERS = {
    "task_created":     lambda e: try_assign(e.task),
    "lock_released":    lambda e: [notify_waiting_agent(e.file), retry_pending(e.file)],
    "lock_expired":     lambda e: force_release(e.file) + notify_holder(e.holder),
    "heartbeat_lost":   lambda e: mark_agent_dead(e.agent) + release_all_locks(e.agent),
    "worktree_ready":   lambda e: dispatch_agent(e.agent, e.task),
}

scheduler.run()  # 阻塞，纯事件回调
```

**效果**：任务到达 → 分配延迟从 0-10s → <100ms，调度器 CPU 占用从持续轮询 → 几乎为零。

## 优化二：Worktree 池预热复用（P1，任务启动快 3-5 倍）

**问题**：每个任务执行 `git worktree add`，涉及完整元数据创建 + 文件复制，中型项目耗时 5-15 秒。

**方案**：调度器启动时预创建 N 个空 Worktree，任务到达时直接 `checkout -f` 切换，完成后 `reset --hard + clean -fd` 归还池子。

```
class WorktreePool:
    def __init__(self, repo_path, pool_size=4):
        self.available = deque()
        self.in_use = {}
        
        # 预创建池
        for i in range(pool_size):
            wt_path = f".multiagent/worktrees/pool_{i}"
            # 只初始化 git 结构，不 checkout 文件
            subprocess.run(["git", "worktree", "add", "--no-checkout", wt_path, "HEAD"])
            self.available.append(wt_path)
    
    def acquire(self, task_id, base_branch="main"):
        if not self.available:
            # 动态扩容
            wt_path = self._create_one()
        else:
            wt_path = self.available.popleft()
        
        # 快速切换到目标分支（比 worktree add 快 5x+）
        subprocess.run(["git", "-C", wt_path, "fetch", "origin", base_branch])
        subprocess.run(["git", "-C", wt_path, "checkout", "-f", f"origin/{base_branch}"])
        subprocess.run(["git", "-C", wt_path, "checkout", "-b", f"agent/{task_id}"])
        
        self.in_use[task_id] = wt_path
        return wt_path
    
    def release(self, task_id):
        wt_path = self.in_use.pop(task_id)
        # 彻底清理
        subprocess.run(["git", "-C", wt_path, "reset", "--hard", "HEAD"])
        subprocess.run(["git", "-C", wt_path, "clean", "-fd"])
        subprocess.run(["git", "-C", wt_path, "checkout", "--detach", "HEAD"])
        self.available.append(wt_path)
```

**效果**：任务启动延迟从 5-15s → 1-3s（仅 checkout 切换）。

## 优化三：符号级锁（P2，并行度由文件级提升到函数级）

**问题**：文件级锁导致同一文件的不同函数无法并行修改。Agent A 改 `auth.ts::login()`，Agent B 必须等 A 释放才能改 `auth.ts::logout()`。

**方案**：锁粒度从 `文件路径` 细化为 `文件路径#符号名`。分配任务前用 tree-sitter 解析目标文件，只锁任务实际触碰的符号。

```
# 锁文件 key 格式变更
# 原来:  .multiagent/locks/src/auth.ts.lock
# 现在:  .multiagent/locks/src/auth.ts#login.lock
#        .multiagent/locks/src/auth.ts#logout.lock
#        .multiagent/locks/src/auth.ts#validateToken.lock

def resolve_locks_for_task(target_files, task_description):
    """根据任务描述，只锁涉及的具体符号，而非整个文件"""
    locks_needed = []
    
    for file in target_files:
        # 解析 AST，提取所有顶层符号（函数/类/导出）
        symbols = parse_symbols(file)  # 使用 tree-sitter 或 LSP
        
        # 用任务描述匹配实际涉及的符号
        touched = match_touched_symbols(task_description, symbols)
        
        if touched:
            for sym in touched:
                locks_needed.append(f"{file}#{sym}")
        else:
            # 无法确定 → 回退到文件级锁（安全第一）
            locks_needed.append(file)
    
    return locks_needed

def has_conflict(task_a_locks, task_b_locks):
    """两个任务的锁集合是否冲突"""
    # 如果 task_a 锁了文件级 (无 #)，则垄断整个文件
    for lock_a in task_a_locks:
        for lock_b in task_b_locks:
            if lock_a == lock_b:
                return True
            # 文件级锁 vs 该文件下的符号级锁 → 冲突
            if '#' not in lock_a and lock_b.startswith(lock_a):
                return True
            if '#' not in lock_b and lock_a.startswith(lock_b):
                return True
    return False
```

**锁目录结构示例**：

```
.multiagent/locks/
├── src/auth.ts#login.lock         # Agent A 持有
├── src/auth.ts#logout.lock        # Agent B 持有（并行！）
└── src/database.ts.lock           # Agent C 垄断整个文件
```

**效果**：大型单体文件（如 2000 行的 `utils.ts`）可被多个 Agent 同时修改不同函数，并行度显著提升。

## 优化四：自适应 Checkpoint（P1，减少 80% 的 fetch 调用）

**问题**：每次 LLM 调用前都 `git fetch + rev-list + rebase`，但多数时候上游无变化。一个任务 10 次工具调用，9 次 fetch 浪费。

**方案**：只在收到合并事件时才 fetch，平时用本地缓存的上游 HEAD 判断。

```
class AdaptiveCheckpoint:
    def __init__(self):
        self.last_fetch_time = 0
        self.last_upstream_head = None
        self.fetch_interval = 30  # 最小 fetch 间隔（秒）

    def should_check(self):
        """是否需要在本次 LLM 调用前执行 checkpoint"""
        now = time.time()
        
        # 30 秒内不重复 fetch
        if now - self.last_fetch_time < self.fetch_interval:
            return False
        
        return True

    def check_and_rebase(self, worktree_path, task_id):
        if not self.should_check():
            return
        
        # fetch 并比较 HEAD
        subprocess.run(["git", "-C", worktree_path, "fetch", "origin", "main"])
        self.last_fetch_time = time.time()
        
        remote_head = subprocess.check_output(
            ["git", "-C", worktree_path, "rev-parse", "origin/main"]
        ).decode().strip()
        
        if remote_head == self.last_upstream_head:
            return  # 无变化，跳过
        
        self.last_upstream_head = remote_head
        
        # 有变化，执行 rebase（与 v2.0 逻辑相同）
        ahead_behind = subprocess.check_output(
            ["git", "-C", worktree_path, "rev-list", "--count", "HEAD..origin/main"]
        ).decode().strip()
        
        if int(ahead_behind) > 0:
            result = subprocess.run(["git", "-C", worktree_path, "rebase", "origin/main"])
            if result.returncode != 0:
                send_user_notification(f"任务 {task_id} Rebase 冲突！")
                raise RebaseConflictError()

    def on_merge_event(self):
        """当 Socket 推送 merge 事件时，强制下次 fetch"""
        self.last_fetch_time = 0  # 强制刷新
```

**配合 Socket 事件**：当其他 Agent 完成提交并推送到主线时，调度器推送 `merge_completed` 事件，Agent 收到后立即标记需要刷新，而不必等到 fetch 间隔。

## 优化五：语义合并分级处理（P3，减少不必要的 LLM 调用）

**核心原则**：能用确定性规则解决的立刻通过；同函数冲突则**尽可能调用 LLM** 做语义分析，因为同一函数内的逻辑变更需要完整的语义理解，不应尝试启发式规则。

```
def ai_merge(base_file, ours_file, theirs_file):
    """分级合并策略"""
    
    # 从 git merge-file 获取冲突标记
    merge_result = subprocess.run(
        ["git", "merge-file", ours_file, base_file, theirs_file],
        capture_output=True
    )
    
    # === Level 1: 无冲突 → 直接通过 ===
    if merge_result.returncode == 0:
        return True, "自动合并无冲突"
    
    # 有冲突标记，解析冲突块
    conflicts = parse_conflict_blocks(ours_file)
    
    all_resolved = True
    
    for conflict in conflicts:
        # === Level 2: 不同函数/符号的变更 → AST 检查后直接合并 ===
        if not is_same_function(conflict["ours"], conflict["theirs"]):
            # 确认两个变更不触碰同一符号体
            if can_auto_merge(conflict):
                apply_both_changes(conflict)
                continue
        
        # === Level 3: 同函数冲突 → 调用 LLM（尽可能用 LLM） ===
        # 不做启发式尝试，直接交 LLM 做语义理解
        resolution = call_llm_for_merge(
            prompt=f"""
            你是代码合并专家。以下冲突发生在**同一函数内部**。
            
            === BASE（原始版本）===
            {conflict['base']}
            
            === OURS（我们的修改）===
            {conflict['ours']}
            
            === THEIRS（他们的修改）===
            {conflict['theirs']}
            
            要求：
            1. 理解双方的修改意图
            2. 如果两者功能独立且可共存 → 合并保留两者
            3. 如果逻辑互相矛盾 → 分析哪个更合理，只保留更合理的一方
            4. 如果无法判断 → 输出 <<<UNRESOLVABLE>>>，我们会交人工裁决
            
            直接输出合并后的最终代码（不含任何解释）：
            """
        )
        
        if "<<<UNRESOLVABLE>>>" in resolution:
            all_resolved = False
            break  # 有人工介入需求，停止处理
        
        apply_resolution(conflict, resolution)
    
    if all_resolved:
        write_file(ours_file, rebuild_file(conflicts))
        return True, "LLM 语义合并成功"
    else:
        return False, "存在无法自动解决的语义冲突，需人工介入"
```

**LLM 调用优化（辅助）**：

```
# 批量合并：同文件有多个同函数冲突时，一次 LLM 调用处理所有
# 避免逐个 conflict block 调用 → N 次变 1 次

# 结果缓存：对完全相同的(base, ours, theirs)三元组缓存结果
# key = hash(base + ours + theirs)，同模式 5 分钟内复用
merge_cache = LRUCache(maxsize=100, ttl=300)
```

## 优化六：分级 TTL + 主动探活（P1，死锁窗口 300s → 90s）

**问题**：固定 300 秒 TTL + 30 秒心跳，Agent 崩溃后锁最多占用 5 分钟。

**方案**：短初始租约 + 长稳定租约 + 主动探活。

```
class LeaseManager:
    INITIAL_TTL = 60      # 初始租约 60 秒
    STABLE_TTL = 180      # 稳定后 180 秒
    HEARTBEAT_GAP = 25    # 心跳间隔 25 秒（初始阶段）
    STABLE_HEARTBEAT = 60 # 稳定后心跳 60 秒
    PROBE_TIMEOUT = 45    # 超过 45 秒未心跳 → 发送 ping
    
    def acquire(self, file, agent_id, task_id):
        lock = {
            "file": file,
            "holder": agent_id,
            "task_id": task_id,
            "expires_at": now() + self.INITIAL_TTL,  # 初始短 TTL
            "phase": "initial",
            "last_heartbeat": now()
        }
        create_lock_file(file, lock)
        return lock
    
    def renew(self, file, agent_id):
        lock = read_lock(file)
        if lock["holder"] != agent_id:
            raise PermissionError("你不是此锁的持有者")
        
        # 已有 2 次成功续约 → 升级为稳定 TTL
        if lock.get("renew_count", 0) >= 2:
            lock["phase"] = "stable"
            lock["expires_at"] = now() + self.STABLE_TTL
        
        lock["last_heartbeat"] = now()
        lock["renew_count"] = lock.get("renew_count", 0) + 1
        write_lock(file, lock)
    
    def health_check_loop(self):
        """调度器维护的探活循环"""
        while True:
            for lock in list_all_locks():
                idle_time = now() - lock["last_heartbeat"]
                
                if idle_time > self.PROBE_TIMEOUT:
                    # 向 Agent 发送 ping（通过 Socket）
                    response = ping_agent(lock["holder"])
                    if response is None:
                        # Agent 无响应 → 释放锁 + 标记失败
                        force_release(lock["file"])
                        mark_task(lock["task_id"], "failed", "Agent 失联")
                        notify_user(f"{lock['holder']} 已失联，任务 {lock['task_id']} 已终止")
                
                if lock["expires_at"] < now():
                    # 租约到期，强制释放
                    force_release(lock["file"])
                    notify_agent(lock["holder"], "你的租约已过期")
            
            sleep(15)  # 每 15 秒检查一次
```

**时间线对比**：

```
崩溃前: 固定方案崩溃后最多锁 300s → 优化后最多 90s
正常时: 30s 心跳 × N 次 → 60s 心跳 × N/2 次（续约次数减半）
```

## 优化七：合并前沙箱验证（P0，防止合并损坏代码）

**问题**：AI 合并后直接推入主线。语义合并可能产生"看起来合理但编译不过/测试不过"的代码。

**方案**：在合并结果推入主线前，先在临时 Worktree 中执行验证管线。

```
def pre_merge_validation(merged_tree, project_config):
    """合并前 CI 沙箱验证"""
    
    # 从 .multiagent/config.json 读取验证命令
    checks = project_config.get("merge_checks", [
        {"name": "type-check",  "cmd": "npx tsc --noEmit"},
        {"name": "lint",        "cmd": "npx eslint src/ --max-warnings 0"},
        {"name": "build",       "cmd": "npm run build"},
        {"name": "unit-test",   "cmd": "npx vitest run"},
    ])
    
    # 在临时目录执行
```

## 优化八：Agent 熔断器（P2，防止反复失败消耗资源）

**问题**：Agent 反复失败不被隔离，持续消耗算力和 API 费用。

**方案**：每个 Agent 维护失败计数，达到阈值自动降级或隔离。

```
class AgentCircuitBreaker:
    FAIL_THRESHOLD_DEGRADE = 2    # 连续失败 2 次 → 降级
    FAIL_THRESHOLD_QUARANTINE = 5  # 连续失败 5 次 → 隔离
    TASK_FAIL_THRESHOLD = 3        # 同一任务失败 3 次 → 标记 stuck
    
    def __init__(self):
        self.agent_states = {}     # agent_id → {failures, state}
        self.task_states = {}      # task_id → {failures, state}
    
    def on_task_success(self, agent_id):
        if agent_id in self.agent_states:
            self.agent_states[agent_id]["failures"] = 0
            # 成功一次就从 degraded 恢复到 normal
            self.agent_states[agent_id]["state"] = "normal"
    
    def on_task_failure(self, agent_id, task_id, reason):
        # Agent 级别
        agent_state = self.agent_states.setdefault(agent_id, {"failures": 0, "state": "normal"})
        agent_state["failures"] += 1
        
        if agent_state["failures"] >= self.FAIL_THRESHOLD_QUARANTINE:
            agent_state["state"] = "quarantined"
            # 释放该 Agent 所有锁，停止分配任务
            release_all_locks(agent_id)
            notify_user(f"Agent {agent_id} 已隔离（连续失败 {agent_state['failures']} 次）")
        elif agent_state["failures"] >= self.FAIL_THRESHOLD_DEGRADE:
            agent_state["state"] = "degraded"
            # 只分配简单任务（target_files 数量 ≤ 2）
            log(f"Agent {agent_id} 已降级")
        
        # 任务级别
        task_state = self.task_states.setdefault(task_id, {"failures": 0})
        task_state["failures"] += 1
        
        if task_state["failures"] >= self.TASK_FAIL_THRESHOLD:
            task_state["state"] = "stuck"
            notify_user(f"任务 {task_id} 连续失败 {self.TASK_FAIL_THRESHOLD} 次，已标记为 stuck，需人工查看")

    def can_accept_task(self, agent_id, task_complexity):
        state = self.agent_states.get(agent_id, {})
        agent_state = state.get("state", "normal")
        
        if agent_state == "quarantined":
            return False, "Agent 已被隔离"
        if agent_state == "degraded" and task_complexity > 2:
            return False, "Agent 已降级，不接受复杂任务"
        
        return True, "ok"
```

## 优化九：WAL 崩溃恢复（P2，保证状态一致性）

**问题**：调度器崩溃时内存状态丢失，磁盘 JSON 可能不一致（task 标了 in_progress 但锁文件被清理了）。

**方案**：所有状态变更先写 WAL，再改 JSON，重启后重放。

```
# WAL 文件路径: .multiagent/wal/wal_0001.log

# 每条 WAL 记录格式（纯文本 JSONL，追加写入）
# {"ts": 1718000000, "op": "assign", "task": "uuid1", "agent": "agent-1", "files": ["auth.ts"]}
# {"ts": 1718000001, "op": "acquire_lock", "file": "auth.ts", "holder": "agent-1", "expires": 1718000300}
# {"ts": 1718000302, "op": "release_lock", "file": "auth.ts", "holder": "agent-1"}
# {"ts": 1718000303, "op": "complete", "task": "uuid1"}

class WALManager:
    def __init__(self, wal_dir=".multiagent/wal/"):
        self.wal_dir = wal_dir
        self.current_segment = None
        self.seq = 0
        self._rotate_if_needed()
    
    def write(self, op, **kwargs):
        record = {"ts": int(time.time()), "op": op, **kwargs}
        line = json.dumps(record) + "\n"
        self.current_segment.write(line)
        self.current_segment.flush()
        os.fsync(self.current_segment.fileno())  # 确保落盘
    
    def recover(self):
        """调度器启动时执行恢复"""
        pending_ops = self._read_all_uncommitted()
        
        for record in pending_ops:
            if record["op"] == "assign" and not is_task_recorded(record["task"]):
                # task 分配了但 JSON 没写入 → 回滚 task 到 pending
                rollback_task(record["task"])
            
            elif record["op"] == "acquire_lock" and not lock_exists(record["file"]):
                # 锁记录存在但文件没有 → 补充创建
                create_lock(record["file"], record["holder"])
            
            elif record["op"] == "release_lock" and lock_exists(record["file"]):
                # 释放记录存在但锁文件还在 → 补充删除
                force_release(record["file"])
            
            elif record["op"] == "complete" and task_status(record["task"]) != "done":
                # 完成记录存在但任务状态未更新 → 补记
                update_task_status(record["task"], "done")
        
        # 清理已处理的 WAL，标记 checkpoint
        self._write_checkpoint()
```

**执行任何状态变更的统一入口**：

```
def safe_state_change(op, callback, **record_data):
    """所有状态变更都必须经过此函数"""
    wal.write(op, **record_data)  # 1. 先写 WAL
    callback()                     # 2. 再执行实际操作
    wal.write("commit", ref=record_data["ts"])  # 3. 标记完成
```

## 优化十：双模 Quorum 合并（P3，提高合并正确性）

**问题**：单次 LLM 调用决定合并结果，可能受幻觉或随机性影响。

**方案**：对 Level 4 冲突（同函数），用两个不同的模型/配置各自生成合并结果，一致才接受。

```
def quorum_merge(conflict):
    """针对同函数冲突的双模验证"""

    # 主模型：高能力模型，temperature 0（确定性输出）
    primary = call_llm(
        model="claude-opus-4-6",     # 最强模型
        temperature=0.0,              # 完全确定性
        prompt=build_merge_prompt(conflict)
    )

    # 校验模型：另一模型或同模型不同采样，temperature 0.1
    secondary = call_llm(
        model="claude-sonnet-4-6",   # 不同模型
        temperature=0.1,
        prompt=build_merge_prompt(conflict)
    )

    # 比较输出（归一化后）
    if normalize(primary) == normalize(secondary):
        return True, primary, "quorum_passed"

    # 不一致时，用最强模型的结果，但打上标记
    # 同时生成差异报告供人工审核
    diff_report = generate_diff(primary, secondary)
    log_warning(f"Quorum 不一致: {diff_report}")

    return True, primary, f"quorum_diff\n差异报告：\n{diff_report}"

def normalize(code):
    """归一化：去空白、统一缩进、排序 import"""
    return ast.unparse(ast.parse(code))  # AST 往返归一化
```

**仅在任务的关键路径（`merge_strategy = "quorum"` 的任务）启用**，避免双倍 API 开销。

## 优化优先级总览

```
P0 (立即实施):
  ├── 事件驱动替代轮询          # 延迟 10s → <100ms
  └── 合并前沙箱 CI 验证        # 防止合并推坏代码

P1 (短期):
  ├── Worktree 池化             # 任务启动快 3-5x
  ├── 自适应 Checkpoint         # fetch 减少 80%
  └── 分级 TTL + 主动探活       # 死锁窗口 300s → 90s

P2 (中期):
  ├── 符号级锁                  # 同文件不同函数可并行
  ├── Agent 熔断器              # 防止故障扩散
  └── WAL 崩溃恢复              # 状态一致性保障

P3 (长期):
  ├── 语义合并分级 + 同函数倾力 LLM
  └── 双模 Quorum 合并          # 合并正确性验证
```
