### 模块一：原子事务锁（解决“部分锁定”竞态）



**问题**：任务要锁 A、B、C，检查时都空闲，但创建锁文件时 C 被抢走。



**解决方案（两阶段提交 + 文件重命名原子性）**：



1. **目录结构**：

   

   .multiagent/

   ├── staging/          # 预备区域

   └── locks/            # 正式生效区域

   



2. **原子锁定算法（Python 伪代码）**：

   `python`

   import os, uuid

   

   def atomic*acquire(task*id, target*files):*

       # 第一阶段：在 staging 中写入意图

       intent*file = f".multiagent/staging/{task*id}.intent"

       with open(intent*file, 'w') as f:*

           json.dump({"files": target*files, "agent": "pending"}, f)*

       

       # 第二阶段：原子性重命名（跨文件系统通常保证原子性）

       for f in target*files:*

           lock*path = f".multiagent/locks/{f.replace('/', '*')}.lock"

           # 关键：如果文件已存在，os.rename 会报错，捕获异常回滚

           try:

               os.rename(intent*file, lock*path)

           except FileExistsError:

               # 回滚：删除已创建的所有锁文件

               rollback*locks(task*id)

               return False, "冲突，部分文件已被抢占"

       

       # 全部成功，删除临时 intent 文件

       os.remove(intent*file)*

       return True, "全部锁定成功"

### 模块二：语义变更注册表（解决“逻辑耦合”）



**问题**：Agent A 改了 `utils/index.ts` 的导出签名，Agent B 改 `helper.ts` 并调用了旧签名，物理无冲突，逻辑崩溃。



**解决方案（极简 AST 导出监控）**：



1. **数据结构**（`.multiagent/registry.json`）：

   `json`

   {

     "src/utils/index.ts": {

       "exports": ["calculate", "formatDate"],

       "hash": "abc123",

       "modified*by": "agent-001",*

       "updated*at": 123456*

     }

   }

   



2. **调度器检查逻辑（任务分配时触发）**：

   `python`

   def semantic*conflict*check(task):

       for file in task.target*files:*

           # 提取文件所在的模块根目录（例如 file: src/utils/helper.ts -\> 模块: src/utils）

           module*root = os.path.dirname(file)*

           

           # 查询 registry 中该模块下是否有最近（5分钟内）被修改的文件

           recent*changes = get*recent*changes(module*root, time*window=300)*

           

           if recent*changes:*

               # 不阻塞任务，但生成“上下文警告”

               warning = f"[语义警告] {module*root} 模块在 {recent*changes['updated*at']} 被 {recent*changes['agent']} 修改了导出接口。建议先用 git diff 查看变更。"

               

               # 将警告写入任务的元数据中

               append*task*metadata(task.id, "semantic*warning", warning)*

               

               # 同时，将警告写入文件系统的一个 .warning 标记，让 Agent 主动读取

               touch(f".multiagent/warnings/{task.id}.md", content=warning)

   



3. **Agent 侧的强制读取**：Agent 在执行 `write` 或 `replace` 工具前，**必须先检查** `.multiagent/warnings/{task.id}.md` 是否存在。若有，**将警告内容强制注入 System Prompt**，让 AI 知道接口已变。

### 模块三：主动上下文净化器（解决“僵尸上下文”）



**问题**：Agent B 运行了 10 分钟，中间主分支合并了 Agent A 的删除函数操作，B 还蒙在鼓里。



**解决方案（Pre-LLM-Call 强制同步钩子）**：



在 Agent 的每个**工具调用循环**（即每次调用 Claude/Codex 生成代码前）插入如下逻辑：



`python`

def pre*llm*checkpoint(worktree*path, task*id):

    # 1. 快速 fetch 远程（不合并）

    subprocess.run(["git", "-C", worktree*path, "fetch", "origin", "main"])*

    

    # 2. 检查本地分支是否落后于 origin/main

    ahead*behind = subprocess.check*output(

        ["git", "-C", worktree*path, "rev-list", "--count", "HEAD..origin/main"]*

    ).decode().strip()

    

    if int(ahead*behind) \> 0:*

        print(f"[净化] 检测到上游有 {ahead*behind} 个新提交，尝试自动 Rebase...")*

        

        # 3. 尝试自动 Rebase

        result = subprocess.run(

            ["git", "-C", worktree*path, "rebase", "origin/main"],*

            capture*output=True*

        )

        

        if result.returncode != 0:

            # 4. Rebase 冲突 -\> 立即终止，通知人工介入

            send*user*notification(f"任务 {task*id} Rebase 冲突，请人工解决！")*

            # 暂停调度器派发，任务状态改为 failed

            update*task*status(task*id, "blocked*by*rebase")*

            raise RebaseConflictError("冲突无法自动解决")

        

        # 5. 成功 Rebase 后，强制刷新 Agent 的内存提示词（通过追加系统消息）

        append*system*message(f"注意：你的代码基已更新至最新提交。")

### 模块四：轻量级事件推送（解决“轮询滞后”）



**问题**：10 秒轮询导致感知滞后，白白浪费算力。



**解决方案（Unix Domain Socket / 命名管道）**：



1. **建立控制通道**：调度器在启动时创建一个 Unix Socket（`.multiagent/event.sock`）。
2. **Agent 注册监听**：每个 Agent 启动后，连接该 Socket，并发送自己的 Agent ID。
3. **事件广播机制**：

- **锁释放事件**：Agent A 完成 `release_lock` 时，自动向 Socket 写入 `{"event": "release", "file": "auth.ts", "agent": "A"}`。
- **定向推送**：调度器维护一个“文件 -\> Agent”的映射表。当收到 `release` 事件时，调度器查找当前正在等待该文件或有任务涉及该目录的其他 Agent（通过查询其 `task.target_files`），**只向这些 Agent 推送**，而不是群发。



4. **Agent 的事件循环（非阻塞）**：

   `python`

   # 在 Agent 主循环中，使用 select 监听 socket 和标准输入

   import socket, select

   sock = connect*to*socket(".multiagent/event.sock")

   

   def agent*loop():*

       while True:

           # 设置超时 0.5 秒，防止完全阻塞

           rlist, *, * = select.select([sock, sys.stdin], [], [], 0.5)

           if sock in rlist:

               event = sock.recv(1024)

               if "auth.ts" in event:

                   print("[实时感知] 其他 Agent 刚释放了 auth.ts，考虑刷新上下文")

                   # 触发上下文刷新（标记当前 AI 上下文为 dirty）

                   context*dirty = True*

           

           if context*dirty:*

               # 下一次 AI 调用前，强制重读文件

               reload*specific*files(["auth.ts"])

               context*dirty = False*
