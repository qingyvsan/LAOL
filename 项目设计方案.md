

### 方案架构（三大核心组件）



| 组件 | 技术选型 | 职责 |
| --- | --- | --- |
| **任务队列** | 本地 `tasks/` 目录 + JSON 文件 | 持久化存储异步任务，记录状态（`pending/in_progress/done`）和文件锁定清单 |
| **文件锁管理** | `agentlocks` (Node.js) 或自建目录锁 | 提供原子性的文件租约（Lease），Agent 修改前必须 `acquire` |
| **工作区隔离** | Git Worktree + 自定义合并驱动 | 每个 Agent 拥有独立物理目录，最终通过“AI语义合并驱动”汇入主线 |
| **感知桥梁** | 文件监听（Watch） + 定时轮询 | 让 Agent 实时感知 `tasks/` 和锁目录的变化 |



----



### 详细工作流设计



#### 1. 用户发布任务

你只需向 `tasks/` 目录丢入一个 Markdown 或 JSON 文件（例如 `task_001.json`），内容包含任务描述、涉及文件路径和优先级。



#### 2. 智能体认领与锁定（防冲突核心）

- 空闲 Agent 定时扫描 `tasks/`，发现 `pending` 任务后，立即调用 `agentlocks acquire <file_path>`。
- **成功**：获得租约，将任务状态改为 `in_progress`，并在锁文件中记录 Agent ID。
- **失败**（文件已被锁定）：Agent 会**立即放弃认领**，将任务状态回退为 `pending`，并等待下次轮询（可设置冷却时间），**绝不强行覆盖**。



#### 3. 并行修改与实时感知

- **独立工作**：每个 Agent 在自己的 Git Worktree 中修改。
- **实时感知机制（关键）**：
    - **主动轮询**：Agent 在执行任务期间，每 30 秒检查一次锁目录。如果发现其他 Agent 新增了与你当前文件相关的锁，它会自动记录日志并调整策略（如暂停当前操作，等待释放）。
    - **被动监听**：利用文件系统的 `watch` 事件，一旦 `tasks/` 或锁目录有变化，立即触发 Agent 刷新“世界状态”，确保它们知道“谁在干什么”。



#### 4. 提交与“智能合并”

- Agent 完成任务并释放锁后，提交 PR（或直接推送到临时分支）。
- **合并驱动**：采用 ****`git-merge-llm`（AI语义合并驱动）代替 Git 默认的逐行合并。
    - 改不同区域（如函数A和函数B）→ **自动静默合并**。
    - 改同一函数逻辑 → AI 分析意图，若无法安全合并，**自动驳回 PR** 并提示冲突原因，等待你手动介入。

### 目录结构示例



your-project/

├── .multiagent/

│   ├── locks/          # 文件租约存放处（如 auth.ts.lock）

│   ├── tasks/          # 任务队列（task*001.json, task*002.json...）

│   └── state.db        # 轻量 SQLite，记录任务历史（可选）

├── worktrees/          # 每个 Agent 的独立工作目录

│   ├── agent-1/

│   └── agent-2/

└── main-branch/        # 主线代码

### 开发建议（利用现有积木）



如果你决定自己开发，强烈建议**不要从零写**，而是按以下顺序组装：



1.  **核心防冲突**：直接使用 ****`agentlocks`（MIT 协议），它提供了完美的原子性文件锁，无需关心底层竞态条件。
2.  **任务管理**：用 Python 或 Node.js 写一个不到 100 行的调度脚本，负责扫描 `tasks/` 目录并调用 Agent（Claude Code/Codex）的命令行接口。
3.  **实时感知**：Node.js 的 `chokidar` 库或 Python 的 `watchdog` 监控目录变化，轻量且成熟。
4.  **合并驱动**：配置 Git 使用 `git-merge-llm`（需设置环境变量指向你的模型 API）。
